# Masters Track vs Undergrad Track — Experience Gap Report

Read-only UX audit. All findings below are grounded in direct file reads, not assumptions. Line numbers refer to the working tree as of this audit (`data/global-coverage-expansion` branch).

## Executive Summary

The single biggest concrete gap: **the masters onboarding wizard is architecturally trapped inside the persistent app chrome, while undergrad onboarding is not.** In `src/App.tsx`, the undergrad route `/onboarding` (line 174-181) renders `OnboardingRouter` directly under `ProtectedRoute` with no layout wrapper — it owns the full viewport. The masters route is nested: `<Route path="/masters" element={<MastersLayout />}>` (line 195-202) contains `<Route path="onboarding" element={<MastersOnboarding />} />` (line 204) as a child. `MastersLayout.tsx` (lines 55-123, 144-150) renders a fixed 256px sidebar (`fixed inset-y-0 left-0 w-64`), a mobile top bar, and a mobile bottom nav around every child route via `<Outlet />` (line 146) — including onboarding. So even before considering color or animation, masters onboarding physically cannot be full-screen: it always renders inside `<main className="lg:ml-64 min-h-screen ...">` next to a permanent nav rail. This matches the product owner's report exactly and is the root architectural cause of the "unfinished sidebar" feeling — everything else (color, motion, richness) is downstream of this.

Beyond that, the two onboarding implementations are entirely different in visual sophistication: undergrad uses a 7-entry per-step color/theme system with custom SVG components (constellation progress, radar chart, animated ring, full-screen loading/insight overlays); masters uses static single-accent styling with a plain flex progress bar and no custom visualizations.

## Other Page Comparisons

**Application tracker:** `src/pages/Applications.tsx` (630 lines) vs `src/pages/MastersApplications.tsx` (184 lines) — undergrad file is 3.4x longer. A structural grep (`framer-motion|drag|Kanban|useState|Card|grid-template`) returns 22 combined occurrences in `Applications.tsx` vs 4 in `MastersApplications.tsx`, indicating substantially more interactive state and card/grid layout complexity on the undergrad side. This pairing was not read line-by-line (out of scope for this pass), but the line-count and structural-keyword disparity is consistent with the onboarding/dashboard pattern: the undergrad artifact is a fuller build-out, the masters equivalent is a leaner pass.

**Program/college discovery:** there is no standalone masters discovery page — instead, "Discover programs" is a search block embedded directly inside `MastersDashboard.tsx` (lines 285-355: a 4-column filter form plus a flat results list). This is the masters-side counterpart to the (separate, presumably richer) undergrad college search/discovery experience, but because it's folded into the dashboard rather than given its own page, it inherits the dashboard's plainer visual treatment rather than any dedicated discovery-page polish.

## Prioritized List of UI/UX Changes for Parity

1. **[Highest priority — architectural] Take masters onboarding out of `MastersLayout`.** In `src/App.tsx`, move the `onboarding` child route (line 204) out of the `/masters` parent route block (lines 195-212) and register it as a sibling top-level route (like undergrad's `/onboarding` at lines 174-181), wrapped only in `ProtectedRoute`, not `MastersLayout`. This is the fix for the reported "plain sidebar" complaint — no color or animation work will fix a wizard that's rendered next to a permanent nav rail.

2. **[High] Add a per-step color/theme system to `MastersOnboarding.tsx`.** Introduce a `STEP_THEMES`-equivalent array (7 entries matching `STEPS` at MastersOnboarding.tsx:38-46) with distinct `bg`/`accent` pairs, and drive the outer container's `background` (currently the static `S.bg` at line 391) from `STEP_THEMES[step].bg`, mirroring `Onboarding.tsx:1117` (`const theme = STEP_THEMES[step - 1]`) and the `transition:'background 0.5s ease'` pattern at `Onboarding.tsx:2065`.

3. **[High] Replace the flat `StepProgress` bar with an SVG progress treatment.** Rebuild `StepProgress` (MastersOnboarding.tsx:124-140) as an SVG-based indicator modeled on `Constellation` (`Onboarding.tsx:77-121`) — per-step nodes, gradient connecting line, active-step pulse animation, and step labels instead of a single text line.

4. **[Medium] Add step-transition animation.** Wrap `renderStep()` output (MastersOnboarding.tsx:405) in a slide-in/slide-out treatment analogous to `Onboarding.tsx:2076` rather than relying solely on the `fadeUp` keyframe (MastersOnboarding.tsx:28) applied via the `key={step}` remount trick (line 404).

5. **[Medium] Give the final "Review" step a reveal moment.** Replace the static `SummaryRow` table (MastersOnboarding.tsx:368-386) with something structurally like `Onboarding.tsx`'s `LoadingSequence` (928-965) — a brief full-screen scanning/celebration sequence before landing on `/masters` — so completing masters onboarding feels like an event rather than a form submit.

6. **[Medium] Introduce at least one custom data visualization mid-flow.** Undergrad uses `RadarChart` (Onboarding.tsx:666-717) and `GPADisplay` (255-288) to make steps 4-6 feel alive. Masters step 3 ("Standardized Tests", MastersOnboarding.tsx:283-301) and step 4 ("Experience & Research", 302-320) are plain input grids with no visual payoff — a simple readiness/competitiveness gauge (the `ReadinessRing` component already exists in `MastersDashboard.tsx:74-88` and could be reused here) would close this gap cheaply.

7. **[Low] Align typography scale.** Bump the masters page/step headline treatment (currently a flat `fontSize:24` h1 at MastersOnboarding.tsx:396) to include at least one large display-scale moment (undergrad hits 72px/42px at Onboarding.tsx:277/753) — e.g., a large animated readiness percentage on the Review step.

8. **[Low] Dashboard density pass.** Once onboarding parity lands, consider porting 2-3 of `Dashboard.tsx`'s richer widgets to `MastersDashboard.tsx` — specifically the hero-banner-with-greeting pattern (Dashboard.tsx:326-363) and the "do this next" urgency-colored callout (368-387) — to close the secondary (dashboard-level) richness gap noted above.
