# `scraper/sources/masters_enrichment.py` — Full Audit

Every function in the file that generates a value (constant, fallback, baseline,
heuristic, or inference) is classified below. File audited in full (237 lines);
line numbers refer to the patched version in this repo.

| Function | Lines | Generated field | Classification | Why it exists | Survives? |
|---|---|---|---|---|---|
| `infer_stem` | 67-81 | `is_stem_designated` | **Safe derived feature** | Deterministic rule: real CIP code → authoritative DHS STEM prefix list, or keyword match against the real program name. Both inputs are real data; the mapping table (DHS STEM list) is an external authoritative source, not invented. Returns `None` (not a guess) when neither signal matches. | **Kept, unchanged.** |
| `infer_opt_eligible` | 84-91 (was 84-88) | `opt_eligible`, `stem_opt_eligible` | **Safe derived feature** (after fix) | US immigration policy fact: F-1 OPT is available to all US grads regardless of major; STEM OPT extension requires STEM designation. This is a real policy rule, not an estimate — the only issue was **fabrication via type coercion**: `bool(is_stem)` silently converted "STEM status unknown" (`None`) into a **false** "not STEM-OPT-eligible" claim. **Fixed** to return `None` for `stem_opt_eligible` when `is_stem` is `None`, instead of asserting a wrong negative. | **Kept, patched.** |
| `compute_admission_difficulty` | 91-108 (was 91-104) | `admission_difficulty` | **Fabricated score (fixed)** | Started every calculation from a hardcoded `50.0` baseline. When `acceptance_rate`, `avg_gpa`, and `avg_gre_quant` were **all** `None` (real DB state for the entire non-null population of this column before this session's cleanup — 643/648 programs), the function still returned the untouched baseline `50.0`, indistinguishable from a real computed difficulty score. This was the exact violation identified in `synthetic_data_inventory.md` and manually nulled in the DB in the prior session. **Fixed**: now returns `None` when zero real inputs exist, matching the already-correct pattern in `compute_roi` (below). | **Kept, patched** — will no longer regenerate the fabricated baseline. |
| `compute_funding_attractiveness` | 110-131 (was 107-128) | `funding_attractiveness` | **Fabricated score (fixed)** | Same shape of bug: started from a hardcoded `0.0` and only added points when real signals existed. With zero real inputs, this silently produced a confident "0% funding attractiveness" — indistinguishable from "verified: this program offers no funding," when the truth was "we know nothing about funding for this program." **Fixed**: now returns `None` when `funding_avail`, `ta`, `ra`, `stipend`, and `full_prob` are all `None`. | **Kept, patched.** |
| `compute_roi` | 134-143 | `roi_score` | **Safe derived feature** (already correct) | Explicitly `return None` when `median_salary`, `tuition_total`, or `program_length_months` is missing — this was already the right pattern before this audit. Used as the reference implementation for fixing the two functions above. | **Kept, unchanged.** |
| `BATCH_SIZE` constant | 143 (was 143) | n/a | **Internal feature only** | Controls commit batching for pooler timeout avoidance — an operational parameter, not a data value, never reaches a user. | **Kept, unchanged.** |
| `enrich_programs` write statement | ~194-216 | writes `is_stem_designated`, `opt_eligible`, `stem_opt_eligible`, `admission_difficulty`, `funding_attractiveness`, `roi_score` | **Safe (structural)** | Uses `COALESCE(is_stem_designated, %(is_stem)s)` and `COALESCE(opt_eligible, %(opt)s)` — preserves any existing non-null value and only fills gaps, appropriate for classification-style inference fields. For `admission_difficulty` and `funding_attractiveness`, the write is an **unconditional overwrite** (not `COALESCE`) — this is intentional and correct now that the compute functions return `None` on empty input: re-running the script after this patch will correctly *re-null* these fields if they somehow got faked again by another code path, rather than leave a stale bad value in place. `roi_score` already used `COALESCE(%(roi)s, roi_score)` to avoid overwriting a good value with a transient `None` (e.g. if salary data is temporarily unavailable in one run) — left as-is since it's the more conservative and already-correct choice for a field with real external salary-source variability. | **Kept, unchanged (aside from now writing `None` instead of fabricated baselines).** |

## Fields NOT generated by this file (verified absent, no action needed)

Grepped the full file for every numeric literal and constant used in scoring
(`STEM_CIP_PREFIXES`, `STEM_KEYWORDS`, `BATCH_SIZE`, the `50.0`/`0.0` baselines above, and
the tuning constants inside each compute function like `* 15`, `* 1.5`, `/ 1500`, `* 0.3`,
`/ 5`). None of the tuning/weighting constants inside `compute_admission_difficulty` or
`compute_funding_attractiveness` are themselves fabricated data — they're modeling
coefficients applied *to* real inputs (same category as the weights in
`consolidatedChancingService.js`'s heuristic engine, which the priority matrix already
treats as a different risk class than literal data fabrication). The bug was never the
coefficients — it was the unconditional baseline that fired when there was nothing to
apply the coefficients to.

## Summary

- **2 functions had a fabrication bug** (`compute_admission_difficulty`,
  `compute_funding_attractiveness`) — both patched to return `None` on empty input,
  matching the already-correct `compute_roi` pattern.
- **1 smaller fabrication-via-coercion bug** (`infer_opt_eligible`'s `stem_opt_eligible`)
  — patched to preserve `None` instead of coercing to `False`.
- **2 functions were already safe** (`infer_stem`, `compute_roi`) — unchanged.
- **Verified via `python -c "import ast; ast.parse(...)"`** — patched file is syntactically valid.
